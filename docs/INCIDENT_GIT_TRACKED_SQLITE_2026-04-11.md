# Инцидент: `magikbook.db` в git и перезапись прод-данных

**Дата:** 2026-04-11 (UTC).  
**Зачем документ:** зафиксировать причину «косяка» с БД и правила, чтобы не повторить.

---

## Что случилось

Файл `magikbook.db` **находился под версионным контролем git** (был закоммичен раньше). В `.gitignore` указано `*.db`, но **правило игнорирования не действует на уже отслеживаемые файлы**.

При подготовке коммита с кодом воркеров выполнили `git checkout HEAD -- magikbook.db`, чтобы «убрать изменения» из рабочей копии. Эта команда **не «отменяет коммит»**, а **перезаписывает файл на диске** содержимым из последнего коммита — то есть **откатило живую SQLite к старому снимку из репозитория**.

Корректное поведение: **не добавлять** `magikbook.db` в коммит (`git add` только нужных путей). Никогда не вызывать `git checkout` / `git restore` на продакшен-БД без явного намерения перезаписать данные.

---

## Восстановление данных

- БД восстановлена с хоста из каталога бэкапов: `data/backups/magikbook/` (см. [`scripts/backup_magikbook_db.sh`](../scripts/backup_magikbook_db.sh)).
- Перед восстановлением API останавливали, затем снова запускали контейнер.

---

## Профилактика (сделано и правила)

1. **`magikbook.db` снят с учёта git** (файл на диске сохранён):  
   `git rm --cached magikbook.db`  
   Коммит в репозитории: `80b3e98` (ветка `main`).

2. **Проверка после клона:** `git ls-files magikbook.db` — должно быть пусто. Если файл снова попал в индекс — не коммитить; при необходимости снова `git rm --cached magikbook.db`.

3. **Запрещено на прод-хосте:**  
   `git checkout -- magikbook.db`,   `git restore magikbook.db` из истории,  
   сброс ветки с «грязным» индексом без проверки списка файлов.

4. Регулярные снимки: раздел «Бэкап `magikbook.db`» в [`PRODUCTION_CHECKLIST.md`](../PRODUCTION_CHECKLIST.md).
